# Frontend Setup Guide: Building a React App for NaCl

**Target audience:** TypeScript developers who have never used React.
**Reference project:** Knife-roll (`ManoloEsS/Knife-roll/knife-roll_frontend/`)
**Goal:** Build a production-quality React frontend for the NaCl password manager.

---

## Step 1: Scaffold the project with Vite

Vite is the standard tool for starting a new React project. It gives you a development server with hot module replacement (instant feedback when you edit code), a build command that bundles everything for production, and sensible defaults for TypeScript, JSX, and CSS.

Use the `create vite` command as described in the Vite documentation, selecting the React + TypeScript template. This generates:

- `index.html` — the HTML shell. Vite injects your bundled JavaScript into this file at build time. You do not edit this file often.
- `src/main.tsx` — the application entry point. This is the first code that runs.
- `src/App.tsx` — the root component. Every component you build nests inside this one.
- `vite.config.ts` — Vite configuration. You will edit this to set up the API proxy and the build output directory.
- `tsconfig.json`, `tsconfig.app.json`, `tsconfig.node.json` — TypeScript configuration. The project-references pattern keeps node-level config (Vite) separate from app-level config (your components).
- `package.json` — dependency manifest. You will add libraries to this file.

**What the config files do:**

- `vite.config.ts` (see Knife-roll: `knife-roll_frontend/vite.config.ts`): The `proxy` option tells the dev server to forward requests matching `/api/` to your backend. The `build.outDir` option tells Vite where to place the compiled output so your backend can serve it as static files. The `base: './'` option ensures asset paths work when served from a subdirectory.
- `tsconfig.json` (see Knife-roll: `knife-roll_frontend/tsconfig.json`): References the app and node configs. This keeps concerns separate — browser code gets JSX and ES2022; build tooling gets Node types and ES2023.

**NaCl-specific note:** NaCl's `vite.config.ts` already proxies `/api/` to `localhost:3333` (the backend port). The build output goes to `../nacl_backend/dist`. Both are correct.

## Step 2: Install dependencies

Runtime dependencies are libraries your app needs in the browser. Dev dependencies are tools needed only during development (testing, linting, formatting).

**Essential runtime dependencies:**

- `react` and `react-dom` — the core library. `react` provides components, hooks, and JSX. `react-dom` renders them into the browser DOM.
- `react-router-dom` — client-side routing. Maps URLs to components so you can navigate without full page reloads.
- `axios` — HTTP client. Makes API calls with a simpler interface than the built-in `fetch`. Supports interceptors (see Step 3).
- `zod` — runtime validation. TypeScript types only exist at compile time. Zod checks data at runtime, so malformed API responses or user input are caught before they cause errors.

**NaCl-specific note:** NaCl has `jsonwebtoken` in its dependencies. This is a Node.js library for signing and verifying JWTs. The browser never needs to sign tokens — it only decodes them. If you need to read the payload of a JWT on the client, install `jwt-decode` instead. Remove `jsonwebtoken` from `package.json`.

See Knife-roll's `knife-roll_frontend/package.json` for the complete dependency list.

## Step 2.5: Map the backend contract before writing any frontend code

The single most important rule of full-stack development: **never write a frontend type without looking at the backend endpoint first.** Every interface you write is a promise about what the server sends and accepts. If that promise is wrong, the app breaks silently (or loudly at runtime).

Before you write any components, services, or validation schemas, read every backend endpoint's request and response shapes. Document them in a single place the frontend can reference.

**When both sides are TypeScript (Knife-roll's case):**

The ideal approach is a shared package. Create a third directory like `packages/shared/` with its own `package.json` and `tsconfig.json`, and use npm workspaces so both the backend and frontend install it. The shared package contains the Zod schemas and inferred types. Both sides import from `@project/shared`. There is exactly one definition of `CreateUserSchema` — the backend uses it to validate requests, and the frontend uses it to validate form input before sending.

If a shared package is not practical (different repos, different deploy cadences), duplicate the schemas but test them against each other. Write a CI step that runs the frontend's Zod schemas against the backend's actual responses in a test environment. If a field is missing, the test fails.

Knife-roll currently duplicates schemas between `src/utils/schemas.ts` (backend) and `src/lib/validation.ts` (frontend). The guide notes this as a practical trade-off for a single-repo project, but the shared-package approach is the recommended pattern for production.

**When the backend is Go and the frontend is TypeScript (NaCl's case):**

You cannot share types across languages. Instead:

1. **Find your source of truth.** In NaCl, it is the Go DTO structs in `nacl_backend/internal/dto/`. Every struct field, its type, its JSON tag, and its validation rule are the contract. Read every DTO file before writing a single frontend type.

2. **Create a contract map.** In a file like `src/api/contract.ts`, document the mapping from each backend endpoint to its frontend types. The map lives next to the API client and is updated whenever either side changes. A typical entry looks like:
   - Backend endpoint: `POST /api/login`
   - Backend DTO (request): `internal/dto/LoginRequest`  
   - Backend DTO (response): `internal/dto/LoginResponse`
   - Frontend Zod schema: `CredentialsSchema` / `LoggedUserSchema`
   - Frontend service: `src/services/login.ts -> login()`

3. **Use `.strict()` on every response schema.** The `strict()` method in Zod rejects any key that is not in the schema definition. If the Go backend adds a new field to a response and the frontend has not updated its schema, the `.parse()` call will throw an error immediately. This catches contract drift at the service layer rather than silently passing unknown data to components. (NaCl already uses `.strict()` in its schemas — this is correct.)

4. **Validate every API response in the service layer.** Every service function must call `.parse()` on the response data before returning it. If the response does not match the schema, the error propagates to the toast handler (set up in Step 11). This means a mismatched contract is always visible to the developer during testing, never silently corrupting the UI.

**Rules of thumb for keeping the contract consistent:**

| Situation | Action |
|---|---|
| Backend adds a field to a response | Add it to the TS Zod schema as optional (`z.string().optional()`), verify it is present in real responses, then make it required |
| Backend removes a field | Remove from the TS Zod schema, then search all components that use that field and remove those references |
| Backend renames a field | Update the TS Zod schema immediately; update the Go DTO JSON tag to alias the old name if clients need migration time |
| Backend changes an endpoint path | Update the URL in exactly one service function |
| A new endpoint is added | Create a new service function and response Zod schema in one commit; add the page that uses it in a separate commit |
| Backend returns an inconsistent shape in error vs success | Define separate Zod schemas for success and error, union them, or use z.discriminatedUnion |

**NaCl-specific contract map:**

| Backend DTO (`internal/dto/`) | Frontend Schema | Endpoint | Service File |
|---|---|---|---|
| `RegisterRequest` | `CredentialsSchema` | `POST /api/users` | `register.ts` |
| `LoginRequest` / `LoginResponse` | `CredentialsSchema` / `LoggedUserSchema` | `POST /api/login` | `login.ts` |
| `CreateVaultEntryRequest` / `VaultEntryResponse` | (needs creation) | `POST /api/vault` | (needs creation) |
| `ListVaultEntriesResponse` | (needs creation) | `GET /api/vault` | (needs creation) |
| `DecryptVaultEntryResponse` | (needs creation) | `GET /api/vault/:id/decrypt` | (needs creation) |
| `UpdateVaultEntryRequest` | (needs creation) | `PATCH /api/vault/:id` | (needs creation) |
| `DeleteVaultEntryResponse` | (needs creation) | `DELETE /api/vault/:id` | (needs creation) |

Each row in this table represents one future service function and one Zod schema pair (request + response). Add rows as the backend grows.

## Step 3: Set up the API client

Every page in your app needs to talk to the backend. Without a centralized client, each page would need to:

1. Hard-code the API base URL
2. Read the JWT from storage and attach it as an `Authorization` header
3. Check for 401 responses and redirect to login

That is repetitive and error-prone. The solution is an axios instance with interceptors.

**Request interceptor (see Knife-roll: `knife-roll_frontend/src/api/client.ts`, lines 5–11):** Before every outgoing request, read the JWT from `localStorage`. If it exists, attach it as `Authorization: Bearer <token>`. This happens automatically for every API call — no page needs to think about auth headers.

**Response interceptor (see Knife-roll: `knife-roll_frontend/src/api/client.ts`, lines 13–22):** After every response, check the status. If it is 401 (unauthorized), clear the stored token and user data from `localStorage`, then redirect to the login page. This handles expired tokens, logged-out sessions, and permission errors in one place.

The client also sets a base URL (`/api`) so every call can be written as `client.get('/schedules')` instead of `axios.get('http://localhost:3001/api/schedules')`.

**NaCl-specific note:** Create this file at `src/api/client.ts`. The interceptors are identical in structure; only the redirect path changes from `/login` to whatever NaCl's login route will be.

## Step 4: Define types and validation schemas from the contract

TypeScript types describe the *shape* of your data at compile time. Zod schemas enforce that shape at runtime. You need both, and both must mirror the backend contract you mapped in Step 2.5.

**Types (see Knife-roll: `knife-roll_frontend/src/services/` files):** Create an interface for every entity the backend sends or receives: `User`, `Credentials`, `PasswordEntry`, etc. Use these as return types for your service functions and as prop types for your components. TypeScript will catch mismatches when you pass the wrong data to a function. Every type should trace back to one row in your contract map.

**Zod schemas (see Knife-roll: `knife-roll_frontend/src/lib/validation.ts`):** Define a schema for every API request body. `z.string().min(1)` ensures non-empty strings. `z.string().email()` validates email format. `z.object({ ... }).refine(...)` validates relationships between fields (e.g., end date after start date). Each schema should match one backend DTO struct's JSON shape exactly.

Use `z.infer<typeof Schema>` to derive TypeScript types from your Zod schemas. This guarantees your types and runtime validation are always in sync — change the schema, and the inferred type changes automatically (see Knife-roll: `LoginInput` derived from `LoginSchema`).

**When backend is TypeScript (shared package):** Both sides import the same file. The backend's Zod schema IS the source of truth. The frontend does not define its own — it imports from `@project/shared`.

**When backend is Go (NaCl):** The frontend manually mirrors every Go DTO. Keep the contract map from Step 2.5 open while you write schemas. Every field in the Go struct should have a corresponding field in the Zod schema. Use `.strict()` to reject unknown keys — this catches when the backend adds a field and the frontend has not updated its schema.

**NaCl-specific note:** NaCl already has `src/schemas.ts` with `CredentialsSchema`, `UserDataSchema`, and `LoggedUserSchema`. These mirror the Go DTOs in `internal/dto/`. The next step is to expand them for every row in the contract map (vault CRUD, account operations) and add inferred types with `z.infer`.

## Step 5: Create the auth context

React state lives inside a component tree. When many components need the same data — like "is the user logged in?" or "who is the current user?" — you would normally pass it down as props through every intermediate component. That gets messy fast.

A context provider is the solution. It makes a value available to every component in its subtree without prop drilling.

**The auth context pattern (see Knife-roll: `knife-roll_frontend/src/context/AuthContext.tsx`):**

1. Define the shape of the context value: `user`, `token`, `login()`, `logout()`, `loading`.
2. Create the context with `createContext`.
3. Build a provider component that uses `useState` to hold `user` and `token`, initializing them from `localStorage` (so the session survives page refresh).
4. The `login` function calls the API, stores the token and user data in both state and `localStorage`, and returns the user object.
5. The `logout` function clears both state and `localStorage`.
6. Wrap the entire app in the provider so every component can call `useAuth()`.

**Why `login` returns the user object (see Knife-roll: `AuthContext.tsx` lines 42–53):** After calling `login`, the component that initiated it usually needs to redirect based on the user's role. If `login` only sets state asynchronously, the component cannot read the new value in the same render cycle. Returning the user object from `login` lets the component navigate immediately.

**Persistence logic (see Knife-roll: `AuthContext.tsx` lines 22–28):** On page refresh, React state resets to the initial value. The `useState` initializer function reads from `localStorage` to restore the previous session. The `useEffect` on mount only fetches the profile if a token exists but no user data is cached — this handles the case where the token is still valid but the user data was cleared.

## Step 6: Build the service layer

Pages should never call `client.get(...)` directly. Every API endpoint should have a corresponding typed service function. This creates a single source of truth for each backend interaction.

**The service pattern (see Knife-roll: `knife-roll_frontend/src/services/` directory):**

- One file per domain: `authService.ts`, `userService.ts`, `scheduleService.ts`, etc.
- Each function has a clear name and typed parameters: `login(email, password): Promise<LoginResponse>`.
- Each function uses the API client from Step 3.
- Return types use the interfaces defined in Step 4.

**Why this matters:**

- If an endpoint URL changes, you edit one service file, not ten pages.
- If the response shape changes, TypeScript flags every file that uses the old type.
- Error handling is centralized: service functions can throw, and the interceptor in Step 3 handles display.
- Pages become presentation logic: they call a service function, get typed data back, and render it.

**NaCl-specific note:** NaCl already has `src/services/login.ts` and `src/services/register.ts`. These follow the service pattern. The next step is to expand this with services for vault operations (save, list, decrypt, delete passwords).

## Step 7: Set up routing

A single-page application needs a client-side router to show different content based on the URL without a full page reload.

**The routing structure (see Knife-roll: `knife-roll_frontend/src/App.tsx`):**

1. Wrap the entire app in `BrowserRouter`. This enables the routing system.
2. Inside, define a `Routes` block containing `Route` elements. Each `Route` maps a path (like `/login`, `/admin`, `/cook`) to a component.
3. Use `Navigate` for redirects (e.g., `/` redirects to `/login` or to the appropriate dashboard based on auth state).
4. Use a catch-all route (`path="*"`) for 404 pages.

**Protecting routes (see Knife-roll: `knife-roll_frontend/src/components/ProtectedRoute.tsx`):**

Not every page should be accessible without authentication. A protected route wrapper checks the auth context and redirects unauthenticated users to the login page. It can also check for specific roles (admin vs cook) and redirect unauthorized users to a different page.

**NaCl-specific note:** NaCl's `App.tsx` currently manages auth state manually with conditional rendering (`user === null ? registerForm : content`). This works for a single page but does not scale. After setting up routing, `App.tsx` becomes a route table. The current auth-gating logic moves into the protected route component. The `NavBar` with `NavLink` components will work once wrapped in `BrowserRouter`.

NaCl's likely routes:
- `/login` — login form (public)
- `/register` — register form (public)  
- `/vault` — password manager dashboard (protected)
- `/account` — account settings (protected)

The empty `src/pages/` directory is where these page components live.

## Step 8: Build reusable components

A component is a function that returns JSX (HTML-like syntax in JavaScript). Components should be small, focused, and reusable.

**Categories of components:**

- **Layout components** (see Knife-roll: `knife-roll_frontend/src/components/Layout.tsx`): A navbar, a sidebar, a main content area. These wrap pages and provide consistent structure. The layout reads auth context to show the username and logout button. It uses `NavLink` (from react-router-dom) for navigation links that highlight the active route.

- **Presentational components** (see Knife-roll: `knife-roll_frontend/src/components/ShiftCard.tsx`): These receive data through props and render it. They have no business logic. The `ShiftCard` receives a `shift` object and an `onUpdate` callback. When the user clicks "Pick Up," it calls a service function, then calls `onUpdate` to refresh the parent's data.

- **Guard components** (see Knife-roll: `knife-roll_frontend/src/components/ProtectedRoute.tsx`): These wrap content and intercept rendering based on conditions. `ProtectedRoute` checks auth state and redirects if the user is not logged in or lacks the required role.

**NaCl-specific note:** NaCl already has `Notification.tsx` (a pure presentational component) and `Register.tsx` (a form component). The `Login.tsx` file is an empty stub and should be removed or replaced by the login page built in Step 9. The `NavBar` needs the router setup from Step 7 to function.

## Step 9: Build pages

A page is a component that represents a full screen. Pages compose smaller components and glue them together with data fetching.

**The page pattern (see Knife-roll: `knife-roll_frontend/src/pages/cook/Dashboard.tsx`):**

1. Use `useState` for local data (the list of shifts).
2. Use `useEffect` to fetch data when the component mounts or when a dependency (like a tab) changes.
3. Call a service function from Step 6 inside the effect. When the data arrives, store it in state.
4. Render the data using the reusable components from Step 8.
5. Handle the loading state: if data is still being fetched, show skeleton placeholders. If data is empty, show an empty state message.
6. Handle the error state: errors from the service call propagate to the axios interceptor, which shows a toast notification (set up in Step 11).

**NaCl-specific pages:**

- `LoginPage.tsx`: Contains the login form. Uses the auth context's `login()` function. On success, redirects to `/vault`. On failure, shows an error message.
- `RegisterPage.tsx`: Contains the registration form. Calls the register service. On success, redirects to `/login`.
- `VaultPage.tsx`: Shows the user's saved passwords. On mount, fetches the password list. Provides a search/filter. Each entry has a show-decrypt, update, and delete action.
- `AccountPage.tsx`: Shows account settings. Allows password change with the old password confirmation pattern.

## Step 10: Add form validation with React Hook Form

Managing form state manually with individual `useState` calls for each field becomes painful as forms grow. You need to track values, errors, touched state, and submission state for every field.

React Hook Form handles all of this. Combined with the `@hookform/resolvers/zod` package, it runs your Zod schemas from Step 4 automatically on every validation trigger.

**The form pattern (see Knife-roll: `knife-roll_frontend/src/pages/LoginPage.tsx`):**

1. Call `useForm<InputType>({ resolver: zodResolver(Schema) })`. The generic parameter gives you type safety across the form. The resolver connects Zod to React Hook Form.
2. Destructure `register`, `handleSubmit`, `formState: { errors, isSubmitting }`.
3. Use `{...register('fieldName')}` spread syntax on each input. This attaches onChange, onBlur, value, and ref automatically.
4. Use `handleSubmit(onSubmit)` as the form's `onSubmit`. It validates with Zod first, then calls your function only if validation passes.
5. Display field errors with `errors.fieldName?.message` next to each input.
6. Use `setError('root', ...)` for API-level errors (wrong password, user not found).

**Why `register()` replaces manual `useState`:** With manual state, you need `const [email, setEmail] = useState('')`, an `onChange` handler, and a validation check on submit. With React Hook Form, `register` handles all of that — value tracking, change events, blur events, and error collection.

**NaCl-specific note:** NaCl currently manages form state manually with `useState` for username and password. After this step, those forms use `useForm` with `CredentialsSchema` from `schemas.ts`. The `RegisterForm` component's props simplify because `register()` and `errors` come from the hook, not from parent state.

## Step 11: Add global error handling and loading states

Without a global error handler, every page needs try/catch blocks around every API call. This is repetitive, and error handling is never consistent.

**Toast notifications (see Knife-roll: `knife-roll_frontend/src/context/ToastContext.tsx`):**

A toast context provider renders a stack of absolutely-positioned notification elements at the bottom-right of the screen. It provides a `showToast(message, type)` function. The axios response interceptor (from Step 3) calls `showToast` on any non-401 error response, extracting the error message from `error.response?.data?.error` or falling back to `error.message`.

The interceptor is connected to the toast context via a callback setter (`setToastHandler`) that is called once when the app mounts. This avoids a circular dependency between the client module and the context.

**Skeleton loading (see Knife-roll: `knife-roll_frontend/src/components/Skeleton.tsx`):**

A skeleton component is a CSS-only placeholder that mimics the shape of content while it loads. `SkeletonCard` shows three short bars with a shimmer animation. `SkeletonTable` shows rows of bars in a table layout. These are shown when `loading` is true, then replaced with the real content when data arrives.

The CSS uses `@keyframes shimmer` and a gradient background with `background-position` animation to create the moving highlight effect.

## Step 12: Add an error boundary

An uncaught JavaScript error in a React component can crash the entire app, showing a white screen with no feedback. An error boundary catches these errors in a specific subtree and renders a fallback UI instead.

**The error boundary pattern (see Knife-roll: `knife-roll_frontend/src/components/ErrorBoundary.tsx`):**

An error boundary is a class component (React does not support error boundaries with hooks). It implements `getDerivedStateFromError` (to update state when an error is thrown) and `componentDidCatch` (for logging). It renders its children normally unless an error is caught, in which case it renders the fallback.

Wrap the route content with `ErrorBoundary`. When any page component throws, the boundary catches it, logs it, and shows a "Something went wrong" message with a "Try Again" button that resets the error state.

## Step 13: Write tests

Vitest is the test runner that ships with Vite. It uses the same transform pipeline as Vite, so you can import your source files directly without configuration.

**What to test:**

1. **Service functions (unit tests):** Mock the axios client with `vi.mock(...)`. Test that each service function sends the correct request and handles the response.
2. **Validation schemas:** Test that valid input passes and invalid input produces the expected error messages. Zod schemas are pure functions — they require no mocking.
3. **Components (integration tests):** Use `@testing-library/react` to render a component, simulate user interactions (typing, clicking), and assert on the rendered output.
4. **Auth context:** Render a component inside the provider and test that `login` and `logout` update the consumed values.

**Knife-roll note:** Knife-roll's testing infrastructure uses Vitest with a setup file at `knife-roll_backend/tests/helpers/setup.ts`. While that example is backend-focused, the vitest configuration (`vitest.config.ts`) and the pattern of test files alongside source files carries over. The frontend vitest config already exists in NaCl's project.

## Step 14: Add real-time updates (optional)

WebSocket lets the server push updates to connected clients without polling. Socket.IO is the most popular WebSocket library because it handles reconnection, room-based messaging, and fallback to HTTP long-polling when WebSocket is unavailable.

**The WebSocket pattern (see Knife-roll: `knife-roll_frontend/src/hooks/useSocket.ts`):**

- The hook receives an `onEvent` callback.
- It connects to the Socket.IO server on mount, passing the JWT in the `auth` option for authentication.
- It subscribes to specific events (`shift:updated`, `shift:pickup`, etc.) and forwards them to the callback.
- On unmount, it disconnects the socket.
- The token is the dependency — when the user logs out, the token becomes null, and the effect cleans up.

In the consuming component (like the cook dashboard), the socket hook triggers state updates when events arrive, so the UI reflects changes without manual refresh.

**NaCl-specific use case:** When another device adds, updates, or deletes a password in the vault, a `vault:updated` event could notify all connected clients to refresh their password list.

## Common mistakes and how to avoid them

1. **Calling hooks conditionally:** Hooks must be called in the same order on every render. Do not put `useState` or `useEffect` inside `if` blocks, loops, or functions. They must be at the top level of the component.

2. **Forgetting the dependency array in `useEffect`:** If you reference a variable inside `useEffect` but do not list it in the dependency array, the effect will still use the old value when the variable changes. The `react-hooks/exhaustive-deps` lint rule catches this.

3. **Mutating state directly:** Never do `state.push(newItem)` or `state.property = value`. Always use the setter function: `setState(prev => [...prev, newItem])`. React relies on reference equality to detect changes.

4. **Putting everything in one context:** A single auth context is fine. But if you add settings, toast state, and theme to the same context, every consumer re-renders when any value changes. Split concerns into separate contexts (auth, toast, theme).

5. **Not handling the loading state:** APIs take time. If you render `data[0].name` before data arrives, the app crashes. Always handle the initial loading state (null data, skeleton, spinner) and the empty state (no results message).

6. **Letting the contract drift without detection:** When the backend adds, removes, or renames a field, the frontend silently breaks if no one notices. Every Zod response schema should use `.strict()`. Every service function should call `.parse()` on the response. This way, a contract mismatch is never silent — it is an explicit error at the service layer. Write a CI test that calls each backend endpoint and validates the response against the frontend Zod schema. If the test fails, the contract broke.
